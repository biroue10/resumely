Accessibility Statement

ApplyCraft is committed to ensuring digital accessibility for people with disabilities. We continually improve the user experience for everyone and apply relevant accessibility standards.

Last updated: June 28, 2026

Conformance status

ApplyCraft aims to conform to the Web Content Accessibility Guidelines (WCAG) 2.2 Level AA. WCAG defines requirements to make web content more accessible to people with disabilities.

Partially conformant  — some content may not fully meet all WCAG 2.2 AA criteria. We are actively working to address all known gaps.

Technical specifications

ApplyCraft relies on the following technologies for conformance:

  • HTML5 semantic elements (main, nav, aside)
  • WAI-ARIA roles and properties (dialogs, tabs, live regions, labels)
  • CSS (respects prefers-reduced-motion and high-contrast preferences)
  • JavaScript (React 18)

The site is designed to be compatible with the following assistive technologies:

  • Screen readers: NVDA + Firefox, JAWS + Chrome, VoiceOver (macOS/iOS) + Safari
  • Keyboard-only navigation
  • Browser zoom up to 400%

What we have done

  • Skip-to-main-content link at the top of every page
  • Semantic HTML landmarks: <main>, <nav>, <aside>
  • All form inputs have programmatically associated labels (htmlFor)
  • All icon-only buttons have accessible names (aria-label)
  • Error messages are linked to inputs via aria-describedby and marked with aria-invalid
  • Modal dialogs use role="dialog", aria-modal, aria-labelledby, and focus management
  • Escape key closes all modal dialogs
  • Navigation uses aria-current="page" to indicate the active section
  • Status messages are announced via aria-live="polite" regions
  • Formatting toolbar buttons expose their function via aria-label
  • Sidebar toggle has aria-expanded state
  • All decorative emoji are hidden from screen readers via aria-hidden
  • Respects prefers-reduced-motion for all animations and transitions
  • Colour contrast ratios meet WCAG AA (minimum 4.5:1 for text)

Known limitations

Area Limitation Status
PDF export Generated PDFs contain a real, selectable/extractable text layer (not images), which helps applicant tracking systems parse them. However, the PDF is not tagged (no semantic structure), so it is not fully accessible to screen readers. The built-in PDF fonts render Latin-script languages (including accented French, Spanish, German); Arabic and other non-Latin scripts are not rendered in PDF — use DOCX for those. Not yet tagged (a limitation we're transparent about). For Arabic/RTL or non-Latin scripts, export DOCX. For screen-reader use, the "Copy" button copies the resume text.
DOCX export Generated DOCX files may not have complete accessibility tags (headings, reading order). The document is intended for further editing in Word / Google Docs. Planned improvement.
Resume preview The live resume preview canvas is a visual representation. It is marked aria-hidden so screen readers receive form data, not the preview layout. By design.
Focus trap in modals The main dialogs (sign-in, feedback, upload) trap Tab / Shift+Tab within the dialog and restore focus to the trigger on close. A few smaller pop-overs still rely on open/close focus management without a strict loop. Largely implemented; remaining pop-overs to follow.

Report an accessibility issue

If you encounter an accessibility barrier on ApplyCraft, please open a GitHub issue or email hello@applycraft.io. We aim to respond within 5 business days.

Please include in your report:

  • The URL of the page where the issue occurs
  • The assistive technology and browser you are using (e.g., NVDA + Firefox 126)
  • A description of the barrier you encountered
  • Screenshots or screen recordings if possible

Formal complaints

If you are not satisfied with our response, you may contact the relevant national accessibility enforcement authority in your country. For users in the EU, refer to the European Commission Web Accessibility Directive resources.
